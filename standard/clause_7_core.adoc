[[ogcapi-common-core]]
=== OGC API - Common - Part 1: Core

https://docs.ogc.org/is/19-072/19-072.html[OGC API - Common - Part 1: Core] is an approved OGC standard defining common requirements for OGC APIs.
The "Core" requirements class of that standard is a dependency of this OGC API - Styles standard.
Additionally, implementations are strongly recommended to implement the https://docs.ogc.org/is/19-072/19-072.html#rc_landing-page-section["Landing Page" requirements class]
which defines requirements for the Landing Page, Conformance Declaration and API Definition resources.

[[base-resource]]
=== Base resource

[[req_core_base-resource-link]]
[width="90%",cols="2,6a"]
|===
^|*Requirement {counter:req-id}* |*/req/core/base-resource-link*
^|A |The content of any base resource (at path `{baseResource}`) with which styles are associated in the API SHALL include a link to a Styles resource at path `{baseResource}/styles` (link relation type 'http://www.opengis.net/def/rel/ogc/1.0/styles').
|===

[[get_styles]]
=== Fetch styles

This operation returns a list of styles that are currently available for the base resource.

[[req_core_styles-op]]
[width="90%",cols="2,6a"]
|===
^|*Requirement {counter:req-id}* |*/req/core/styles-op*
^|A |The server SHALL support the HTTP GET operation at the path `{baseResource}/styles`.
|===

[[req_core_styles-success]]
[width="90%",cols="2,6a"]
|===
^|*Requirement {counter:req-id}* |*/req/core/styles-success*
^|A |A successful execution of the operation SHALL be reported as a response with an HTTP status code `200`.
^|B |The content of that response SHALL be based upon the following OpenAPI 3.0 schema:

[source,YAML]
----
type: object
required:
  - styles
properties:
  styles:
    type: array
    items:
      type: object
      required:
        - id
        - links
      properties:
        id:
          type: string
        title:
          type: string
        description:
          type: string
        keywords:
          type: string
	attribution:
          type: string
	geoDataClasses:
          type: array
          items:
            type: string
            format: uri
        pointOfContact:
           type: string
        aiAssistance:
           description: Acknowledgement of the use of AI assistance to generate the style
           type: string
        license:
           type: string
        version:
           type: string
        links:
          type: array
          minItems: 1
          items:
            $ref: 'http://schemas.opengis.net/ogcapi/features/part1/1.0/openapi/schemas/link.yaml'
  default:
    type: string
----
^|C |The `styles` member SHALL include one item for each style currently on the server.
^|D |The `id` member of each style SHALL be unique.
^|E |Each style SHALL have at least one link to a style encoding supported for the style (link relation type: `stylesheet`) with the `type` attribute stating the media type of the style encoding.
^|F |The `default` member SHALL, if provided, be the `id` of one of the styles in the `styles` array.
^|G |If a schema is available for the data for which the style is intended, a link to that schema using the link relation `\http://www.opengis.net/def/rel/ogc/1.0/schema` SHALL be included in the Styles resource.
^|H |If a thumbnail is available for a style in the style metadata, a link with the link relation type `preview` SHALL be provided in the Styles resource. 
|===

The metadata, including the links to the schema and previews, should also be embedded within the representation of the style, if that is supported by the style encoding.
Also including this metadata in the Styles resource provide clients access to this information without the need to send requests for each all style resource for discovering
a specific style of interest.

[[rec_core_style-title]]
[width="90%",cols="2,6a"]
|===
^|*Recommendation {counter:rec-id}* |*/rec/core/style-title*
^|A |If a style has a title, it SHOULD be included in the `title` member of the style object.
|===

[[get_style]]
=== Fetch style

This operation returns the stylesheet of a style.

[[req_core_style-op]]
[width="90%",cols="2,6a"]
|===
^|*Requirement {counter:req-id}* |*/req/core/style-op*
^|A |The server SHALL support the HTTP GET operation at the path `{baseResource}/styles/{styleId}` for each style referenced from the Styles resource at `{baseResource}/styles`.
|===

[[req_core_style-success]]
[width="90%",cols="2,6a"]
|===
^|*Requirement {counter:req-id}* |*/req/core/style-success*
^|A |A successful execution of the operation SHALL be reported as a response with an HTTP status code `200`.
^|B |The content of that response SHALL conform to the media type stated in the `Content-Type` header.
^|C |The language used in linguistic text in the response SHALL be consistent with the language stated in the `Content-Language` header.
|===

[[req_core_style-feature-type]]
[width="90%",cols="2,6a"]
|===
^|*Requirement {counter:req-id}* |*/req/core/style-feature-type*
^|A a|The feature type identifiers qualifying a styling rule in the style sheet response SHALL resolve to the data in an OGC API collection by matching in this strict order of priority:

1. An explicit feature type name identified at the individual feature level such as
data layers in multi-layer vector tiles (e.g., layer `name` in https://github.com/mapbox/vector-tile-spec/tree/master/2.1#41-layers[Mapbox Vector Tiles]),
https://docs.ogc.org/is/21-045r1/21-045r1.html#feature-types[JSON-FG],
or a schema https://docs.ogc.org/is/23-058r2/23-058r2.html#_type[role].
2. The single feature type identifier listed in the `featureTypes` array of the collection description.
3. The `{collectionId}` of the collection resource itself.
|===
